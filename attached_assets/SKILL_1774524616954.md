# Objetivo

Transformar insumos técnicos brutos em documentação semântica, detalhada e reutilizável, pronta para colar no Outline.

## Fluxo principal

1. Identificar o módulo e o macro-contexto de negócio.
2. Ler a resolução técnica inteira antes de estruturar a resposta.
3. Separar fatos confirmados de inferências.
4. Produzir a documentação final no formato do template em `references/output-template.md`.
5. Se houver menções a telas, campos, mensagens, botões ou estados de interface, complementar com a seção opcional de microcopy.
6. Se houver menção a processamento documental, OCR, extração, redação de PII, assinatura, formulários PDF ou conversão de arquivos, adicionar a seção opcional de orientação da Nutrient DWS Processor.

## Regras de comportamento

- Inferir o módulo quando ele não vier explicitamente informado e declarar a inferência logo no início.
- Não inventar nomes de tabelas, funções, endpoints, campos ou integrações. Quando o insumo não trouxer evidência suficiente, usar `não informado`, `não especificado` ou `inferência`.
- Priorizar explicação de regra de negócio e impacto sistêmico antes de repetir o texto do card.
- Traduzir linguagem de desenvolvimento para linguagem de documentação técnica sem perder precisão.
- Explicitar risco operacional, dependências e pontos de quebra quando houver input inesperado.
- Manter tom profissional, técnico e detalhista.
- Entregar texto pronto para colar no Outline, sem prefácio conversacional e sem metacomentários.

## Estrutura obrigatória da saída

Sempre gerar, nesta ordem:

1. `Contexto do módulo`
2. `Objetivo de negócio`
3. `Mapeamento de regras de negócio`
4. `Dicionário de interface e campos`
5. `Análise de impacto (developer view)`
6. `Histórico de mudança`
7. `Integrações e apis`
8. `Critérios de atenção, riscos e validações`

Adicionar somente quando houver evidência suficiente:

9. `Microcopy e i18n`
10. `Orientação de processamento documental com nutrient dws processor`

## Regras por seção

### 1) Contexto do módulo

- Identificar o macro-contexto: financeiro, fiscal, estoque, compras, vendas, cadastros, documentos, integrações, segurança, relatórios ou outro equivalente.
- Explicar em 1 a 3 parágrafos qual problema de negócio a funcionalidade atende.
- Se o card for técnico demais, traduzir a intenção funcional para impacto de negócio.

### 2) Objetivo de negócio

- Resumir em bullets curtos:
  - dor ou necessidade
  - comportamento esperado do sistema
  - resultado percebido pelo usuário

### 3) Mapeamento de regras de negócio

- Descrever a lógica do sistema em passos numerados.
- Cobrir gatilho, validações, decisão, persistência, retorno visual e exceções.
- Sempre incluir:
  - condição de entrada
  - processamento
  - condição de bloqueio ou falha
  - resultado final

### 4) Dicionário de interface e campos

- Usar tabela com as colunas exatas:
  - `Nome do campo`
  - `Tipo`
  - `Descrição`
  - `Regra/Opções`
- Se a resolução não listar campos, registrar os campos inferíveis a partir do fluxo e marcar claramente o que é inferido.
- Tipos permitidos: `Texto`, `Data`, `Número`, `Booleano`, `Seleção`, `Arquivo`, `Tabela`, `Oculto`, `Não informado`.

### 5) Análise de impacto (developer view)

Cobrir de forma objetiva:

- funções, rotinas, jobs, serviços ou módulos potencialmente afetados
- tabelas, entidades ou documentos que podem sofrer leitura/escrita
- pontos onde o sistema pode quebrar com input inesperado
- impactos em navegação, permissões, integrações, logs, auditoria e performance

Quando a resolução técnica não trouxer nome técnico real, descrever por categoria funcional, por exemplo: `rotina de validação`, `endpoint de persistência`, `processo de cálculo`, `integração de assinatura`.

### 6) Histórico de mudança

Usar sempre uma tabela comparativa com estas colunas:

- `Aspecto`
- `O que era`
- `O que é agora`
- `Impacto direto para o usuário`

Traduzir a resolução técnica para diferença observável no sistema.

### 7) Integrações e apis

- Listar sistemas externos citados ou inferíveis com segurança.
- Informar se a funcionalidade consome, envia, consulta, assina, valida ou sincroniza dados.
- Se não houver integração, escrever explicitamente `não há integração externa evidenciada no insumo`.

### 8) Critérios de atenção, riscos e validações

Incluir checklist curto com:

- validações obrigatórias
- riscos de regressão
- dependências implícitas
- cenários de teste recomendados

## Seção opcional: microcopy e i18n

Adicionar apenas se o insumo contiver textos de interface, mensagens, botões, modais, banners, estados vazios, placeholders, tooltips ou erros.

### Modo escrever

Para cada item, gerar de 1 a 3 opções ranqueadas em tabela com:

- `Superfície`
- `Texto sugerido`
- `Chave i18n sugerida`
- `Justificativa`

### Modo revisar

Apontar:

- inconsistências de tom e voz
- verbos pouco acionáveis
- ambiguidade
- termos difíceis de localizar
- chaves i18n frágeis ou pouco escaláveis

Priorizar frases curtas, claras, focadas em ação e localizáveis.

## Seção opcional: orientação de processamento documental com nutrient dws processor

Adicionar apenas se o caso mencionar documentos, pdf, docx, html, imagens, ocr, extração, redação de pii, assinatura digital, marca d'água ou formulários pdf.

### Escopo desta seção

- orientar o uso da API
- não afirmar que executou chamadas reais
- não prometer resultados de API sem entrada concreta

### Conteúdo esperado

1. objetivo do uso da API naquele fluxo
2. etapa do processo onde a API entra
3. operação sugerida entre:
   - conversão de formato
   - extração de texto/tabelas
   - exportação para txt/xlsx
   - ocr multilíngue
   - redação de pii
   - marca d'água
   - assinatura digital
   - preenchimento de formulário pdf
4. exemplo de pipeline em json, quando isso ajudar
5. cuidados de conformidade, auditoria e qualidade do input

## Regras de formatação para Outline

Seguir as orientações resumidas em `references/outline-formatting.md`.

Aplicar estas convenções:

- usar headings semânticos; não usar decoração textual para simular hierarquia
- usar tabelas apenas quando houver comparação, inventário ou dicionário de campos
- usar listas numeradas para fluxos e listas simples para checklists
- usar notices apenas para alertas, observações e riscos relevantes
- usar LaTeX somente quando houver fórmula real; manter compatibilidade com KaTeX/Outline
- não usar excesso de emojis, adornos ou blocos visuais supérfluos

## Estratégia para lacunas no insumo

Quando faltarem detalhes, seguir esta ordem:

1. extrair o máximo possível do texto fornecido
2. sinalizar premissas explicitamente
3. entregar uma versão utilizável mesmo incompleta
4. listar no fim um bloco curto de `informações ainda necessárias` apenas se isso realmente bloquear precisão adicional

## Template obrigatório

Usar o template base em `references/output-template.md`.
